Product experience report

From ad-hoc prompting to repeatable Agentic Loop Working.

This report explains how ZAgenticLoop should land for a new user: what problem it solves, which mental model to learn first, which patterns to trust, and how the tool family lowers the cost of adopting loops without hiding the safety work.

Product thesis: ZAgenticLoop is not another agent prompt collection. It is a reference system for designing loops that schedule work, preserve memory, verify output, control budgets, and hand off judgment to humans at the right boundary.

User story map

The user experience should help people choose the smallest useful loop, run it in report-only mode, then graduate only after state, verification, and budget signals are visible.

New adopter

I want my first safe loop in five minutes.

Start with Daily Triage, create STATE.md, run L1 report-only, and learn the loop's failure modes before allowing edits.

Maintainer

I want recurring triage without losing judgment.

Use Pattern, Skill, State, Run Log, Budget, and Human Gate language to keep recurring work auditable.

Team lead

I want confidence before unattended automation.

Use Loop Readiness Score, verifier separation, worktree evidence, and cost observability before L3.

Platform builder

I want stable semantics across CLIs and MCP.

Build on @jununfly/zj-loop-core, registry facts, semantic queries, and read-only MCP surfaces.

Goal operator

I want bounded run-until-done readiness.

Use zj-goal-audit separately from loop readiness when the work has an explicit stopping condition.

Experience rule

Teach calibration before autonomy.

The product should make L1 feel like the normal first success, not a less impressive version of L2 or L3.

Backbone design

The core mental model is a pipeline from reusable operating model to real-world proof. This is the spine README, Quickstart, starters, audit tooling, and stories should keep repeating.

PatternGoal, cadence, risk, phases, human gates, and cost metadata live in patterns/registry.yaml.
StarterCopyable scaffold turns a pattern into skills, route table, state, budgets, run logs, and first command text.
MemoryState and Run Log keep facts outside any one agent conversation and make loop activity inspectable.
VerifierMaker/checker split, worktrees, and readiness gates prevent the agent from grading its own work.
StoryProven Usage and failure stories calibrate trust, costs, escalation rules, and graduation criteria.

Technical advantages

ZAgenticLoop's strongest product advantage is that it turns loop design into shared semantics instead of scattered instructions.

Registry-first patterns

Pattern facts are machine-readable, validated, and reused by audit, cost, docs, init, and MCP surfaces.

Tool-agnostic intent

Patterns describe operating models; examples and starters map them to Grok, Claude Code, Codex, OpenClaw, and GitHub Actions.

Core-owned semantics

@jununfly/zj-loop-core centralizes registry loading, semantic queries, project evidence, and shared CLI harness behavior.

Readiness as product UX

zj-loop-audit converts abstract safety guidance into a score, level, findings, and concrete next steps.

Cost before cadence

zj-loop-cost makes token budget visible before a user schedules high-frequency loops.

Read-only MCP bridge

The MCP server exposes local loop knowledge to agents while keeping semantic policy in core and side effects outside the protocol adapter.

Core feature map

The feature set should be presented as a learning path: initialize, estimate, audit, synchronize, integrate, then handle goal-style work separately.

Feature User value Best first use
zj-loop-init Creates starter files, route table, state, budget, run log, and first command for a chosen pattern. Use Daily Triage with a Codex, Grok, or Claude Code starter.
zj-loop-cost Estimates token spend by pattern, level, and cadence before scheduling. Compare Daily Triage at L1 with CI Sweeper at a 5-15 minute cadence.
zj-loop-audit Scores Loop Readiness and suggests concrete upgrades. Run after scaffold and after the first state update.
zj-loop-sync Detects drift between loop configuration and current project evidence. Use after editing loop config, budget, or state conventions.
zj-loop-mcp-server Gives agents read-only access to patterns, skills, route table, state, safety docs, and semantic loop tools. Use when an agent needs to inspect loop knowledge without duplicating docs in prompt text.
zj-goal-audit Audits bounded run-until-done goal readiness as a sibling package. Use for explicit goals with stop conditions, not recurring cadence loops.
zj-loop-core Shared domain kernel for registry, project evidence, semantic queries, and CLI harness. Use when building a new product surface or adapter over loop knowledge.

Pattern showcase

The pattern catalog should be shown as a risk-aware menu, not as a flat list. Users need to see cadence, risk, week-one posture, token cost, and starter path together.

Pattern Best for Cadence Week one Cost
Daily Triage Lowest-risk first loop and recurring project scan. 1d-2h L1 report Low
Changelog Drafter Release-note drafts with human review. 1d or tag L1 draft Low
Issue Triage Deduplicate, prioritize, and propose labels. 2h-1d L1 propose-only Low
Roadmap-Sliced Development Ambiguous product, architecture, docs, and release initiatives. 1d L2 guided Medium
Post-Merge Cleanup Off-peak cleanup after merges reach main. 1d-6h L1 off-peak Low
Dependency Sweeper Patch-only dependency updates with verifier and gates. 6h-1d L2 patch-only Medium
PR Steward Shepherd PRs through review, CI, rebase, and merge readiness. 5m-15m L1 watch High
CI Sweeper React to failing CI with minimal fixes and escalation. 5m-15m L2 cautious Very high

Case evidence

Stories are part of the product surface because they teach the limits. They reduce learning friction by showing what worked, what broke, and what changed in the operating model.

Failure story

Why We Killed Our CI Sweeper

Jumping straight to auto-fix created runaway spend and noisy PRs. The fix was report-only first, verifier separation, branch allowlist, and daily budget.

Graduation story

L1 to L2 Graduation

Two weeks of L1 calibrated priority judgment before small assisted fixes. The story turns readiness into concrete adoption criteria.

Low-risk win

Changelog Drafter Week One

Drafting release notes cut human review time while keeping publish behind a Human Gate.

Coordination lesson

Multi-Loop Collision

Multiple loops require branch locks, collision detection, and explicit ownership boundaries before they become invisible background work.

Recommended learning funnel

The public README should keep this order visible. It builds user confidence while making the safety model feel like the product, not a warning label.

Step 1

Quickstart

Scaffold Daily Triage and run report-only. The first win is a correct STATE.md, not an automated PR.

Step 2

Pick by pain

Use the Pattern picker to choose by risk, cadence, token cost, and team need.

Step 3

Audit readiness

Run audit and cost tools before increasing cadence or moving from L1 to L2.

Step 4

Calibrate with stories

Read success and failure cases before unattended operation. They encode practical Human Gate and Escalation rules.
