Product experience report

从 ad-hoc prompting 走向可重复的 Agentic Loop Working。

这份报告说明 ZAgenticLoop 应该如何进入新用户心智：它解决什么问题，用户应该先学哪个模型，哪些 patterns 值得信任，以及工具族如何在不隐藏安全工作的前提下降低采用成本。

产品论点：ZAgenticLoop 不是另一个 agent prompt 集合。它是一套参考系统，用来设计能调度工作、保存 memory、验证输出、控制 budget，并在正确边界把判断权交回给人的 loops。

User story map

产品体验应该帮助用户选择最小可用 loop，先以 report-only 模式运行，然后只在 state、verification 和 budget 信号可见之后再升级。

New adopter

我想在 5 分钟内跑起第一个安全 loop。

从 Daily Triage 开始，创建 STATE.md，先跑 L1 report-only，在允许编辑前理解 loop 的 failure modes。

Maintainer

我想要 recurring triage，但不想失去判断权。

用 Pattern、Skill、State、Run Log、Budget 和 Human Gate 这套语言，让重复工作保持可审计。

Team lead

我想在 unattended automation 前获得足够信心。

在 L3 前使用 Loop Readiness Score、verifier separation、worktree evidence 和 cost observability。

Platform builder

我想让 CLIs 和 MCP 共享稳定语义。

基于 @jununfly/zj-loop-core、registry facts、semantic queries 和 read-only MCP surfaces 构建。

Goal operator

我想审计 bounded run-until-done work。

当工作有明确 stop condition 时，使用 zj-goal-audit，不要把它混同于 recurring loop readiness。

Experience rule

先教 calibration，再谈 autonomy。

产品应该让 L1 成为正常的第一步成功，而不是 L2/L3 的低配版本。

骨干设计

核心心智模型是一条从 reusable operating model 到 real-world proof 的管线。README、Quickstart、starters、audit tooling 和 stories 都应该反复强化这条主干。

PatternGoal、cadence、risk、phases、human gates 和 cost metadata 存在 patterns/registry.yaml 中。
Starter可复制 scaffold 将 pattern 变成 skills、state、budgets、run logs 和 first command text。
MemoryState 和 Run Log 把事实保存在单次 agent 对话之外，让 Loop Activity 可检查。
VerifierMaker/checker split、worktrees 和 readiness gates 避免 agent 自己给自己打分。
StoryProven Usage 和 failure stories 校准 trust、cost、escalation rules 和 graduation criteria。

技术优势

ZAgenticLoop 最强的产品优势，是把 loop design 变成共享语义，而不是散落在各处的 prompt 和说明。

Registry-first patterns

Pattern facts 是机器可读、可验证的，并被 audit、cost、docs、init 和 MCP surfaces 复用。

Tool-agnostic intent

Patterns 描述 operating models；examples 和 starters 再映射到 Grok、Claude Code、Codex、OpenClaw 和 GitHub Actions。

Core-owned semantics

@jununfly/zj-loop-core 统一 registry loading、semantic queries、project evidence 和 CLI harness behavior。

Readiness as product UX

zj-loop-audit 把抽象安全建议转成 score、level、findings 和具体下一步。

Cost before cadence

zj-loop-cost 让用户在调度高频 loops 前看见 token budget。

Read-only MCP bridge

MCP server 暴露本地 loop knowledge 给 agents，同时把 semantic policy 留在 core，把 side effects 留在 protocol adapter 之外。

核心 feature map

Feature set 应该按学习路径呈现：initialize、estimate、audit、synchronize、integrate，然后单独处理 goal-style work。

Feature用户价值最佳首次使用方式
zj-loop-init为选定 pattern 创建 starter files、state、budget、run log 和 first command。用 Codex、Grok 或 Claude Code starter 跑 Daily Triage。
zj-loop-cost按 pattern、level 和 cadence 在调度前估算 token spend。比较 Daily Triage L1 与 5-15 分钟 cadence 的 CI Sweeper。
zj-loop-audit计算 Loop Readiness 并给出可执行升级建议。scaffold 后运行一次，第一次 state update 后再运行一次。
zj-loop-sync检测 loop configuration 与当前 project evidence 之间的 drift。修改 loop config、budget 或 state conventions 后使用。
zj-loop-mcp-server让 agents 只读访问 patterns、skills、state、safety docs 和 semantic loop tools。当 agent 需要检查 loop knowledge，但不想把 docs 复制到 prompt 中时使用。
zj-goal-audit作为 sibling package 审计 bounded run-until-done goal readiness。用于有明确 stop condition 的 goals，而不是 recurring cadence loops。
zj-loop-coreregistry、project evidence、semantic queries 和 CLI harness 的共享 domain kernel。构建新的 product surface 或 loop knowledge adapter 时使用。

案例证据

Stories 是产品界面的一部分，因为它们教会用户边界在哪里。它们通过展示 worked、broke 和 changed，降低学习门槛。

Failure story

Why We Killed Our CI Sweeper

直接跳到 auto-fix 导致 runaway spend 和 noisy PRs。修正方式是先 report-only、verifier separation、branch allowlist 和 daily budget。

Graduation story

L1 to L2 Graduation

两周 L1 校准了 priority judgment，再开启 small assisted fixes。这个 story 把 readiness 变成具体采用标准。

Low-risk win

Changelog Drafter Week One

Draft release notes 降低了 human review time，同时把 publish 留在 Human Gate 后面。

Coordination lesson

Multi-Loop Collision

多个 loops 需要 branch locks、collision detection 和明确 ownership boundaries，才能成为可靠的 background work。

推荐学习漏斗

公开 README 应该持续露出这个顺序。它能建立用户信心，并让 safety model 成为产品本身，而不是贴在旁边的警告。

Step 1

Quickstart

Scaffold Daily Triage 并 run report-only。第一个胜利是正确的 STATE.md，不是自动 PR。

Step 2

Pick by pain

用 Pattern picker 按 risk、cadence、token cost 和 team need 选择。

Step 3

Audit readiness

在提高 cadence 或从 L1 走向 L2 之前，运行 audit 和 cost tools。

Step 4

Calibrate with stories

在 unattended operation 前阅读成功和失败案例。它们编码了实用的 Human Gate 和 Escalation rules。
